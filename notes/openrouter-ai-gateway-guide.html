01背景：为什么直连 Gemini API 会失败

前几天在客户端里配 Gemini，踩了一圈坑，过程挺典型，先复盘一下，方便对照自己卡在哪一步。

第一步：模型名填成了展示名

在模型设置里填了 Gemini 3.1 Flash Lite——这是控制台里的展示名称，不是 API 模型 ID。API 要的是全小写、连字符格式，比如 gemini-3.1-flash-lite。填错名字，网关找不到对应模型，直接抛 502。

第二步：改成 API ID 还是 502

把名字改对之后，依然 502。这时候就不是配置问题了——浏览器直接访问 generativelanguage.googleapis.com 显示 404（其实正常，根路径没页面），但客户端里请求 /v1beta/openai/chat/completions 就是不通。

第三步：代理全局模式也不行

把代理切成「全局」，YouTube 能开，但 Gemini API 还是 502。说明问题不在路由规则，而在代理节点本身到不了 Google API 域名。常见三种情况：

现象根因判断依据
YouTube 能开，Google API 不通节点只解锁流媒体，不开放 Google Cloud / API 域名全局模式仍失败
改了模型名仍 502请求没真正到达 Google 后端（被拦截 / 污染）不是 401/404，而是 502
本地 DNS 解析到错误 IPDNS 污染，需开启远程 DNS 解析全局模式 + 远程 DNS 后仍不行 → 排除此项
⚠️ 关键判断

502 不是本地网络问题，也不是 API Key 错误（Key 错会报 401/403）。它意味着请求发出去了，但没到达 Google 后端。在你这个场景里，最大概率是代理节点对 generativelanguage.googleapis.com 这个域名不通。

02什么是 OpenRouter

OpenRouter 是一个统一的 AI 模型 API 网关。它把市面上几百个模型收进一个 OpenAI 兼容的端点，你用一套代码或客户端就能调所有模型。

它覆盖的厂商包括：OpenAI、Anthropic、Google、Meta、Mistral，以及大量开源模型（DeepSeek、Qwen、Llama 等）。核心特点：

  • 单一端点：https://openrouter.ai/api/v1，完全兼容 OpenAI SDK。
  • 一个 Key 通吃：在 OpenRouter 注册一次，拿到 sk-or-v1-... 的 key，所有模型都能用。
  • 换模型只改名字：想从 Gemini 换成 Claude，改一个 model 字段即可，不用改代码、不用换 key。
  • 按 token 计费：有免费模型和付费模型，先充值 credit，调用扣费。

03为什么它能解决国内访问问题

关键不在「它更快」，而在请求路径变了。

直连模式：你的客户端 → 直接请求 generativelanguage.googleapis.com（国内常不通）
OpenRouter 模式：你的客户端 → 请求 openrouter.ai → OpenRouter 的服务器（在美国）再去请求 Google / Anthropic

也就是说，你不再需要直连 Google 的 API 域名。只要你的网络能到 openrouter.ai（这个域名相对 Google Cloud API 更容易连通，很多代理节点对其也更友好），就能间接用上 Gemini、Claude 等所有模型。

一句话：OpenRouter 帮你「代请求」。Google 接口连不上，就让 OpenRouter 的服务器去连——你只跟 OpenRouter 打交道。

04实战：在客户端里配置

以任意 OpenAI 兼容客户端（Cherry Studio、各类对话 App）为例，步骤很简单：

1. 注册并拿 Key

去 openrouter.ai 注册，在 API Keys 页面生成一个 key，格式是 sk-or-v1-xxxxxxxx。

2. 添加自定义提供方

字段填什么
名称OpenRouter（随便起）
API 地址（Base URL）https://openrouter.ai/api/v1
API Keysk-or-v1-xxxxxxxx

3. 添加模型

模型名填 OpenRouter 的 slug，例如：

  • google/gemini-2.5-pro
  • anthropic/claude-3.7-sonnet
  • openai/gpt-4o

保存后发一条测试消息即可。

4. 代码调用示例（Python）

如果你是自己写程序，直接复用 OpenAI SDK，只改 base_url 和 api_key：

from openai import OpenAI

client = OpenAI(
    base_url="https://openrouter.ai/api/v1",
    api_key="sk-or-v1-xxx",   # 换成你的 key
)

resp = client.chat.completions.create(
    model="google/gemini-2.5-pro",
    messages=[{"role": "user", "content": "用一句话解释什么是 API 网关"}],
)
print(resp.choices[0].message.content)

就这么几行，原来调 OpenAI 的代码几乎不用改，模型却能从 GPT 换成 Gemini / Claude。

06费用与配额

  • 免费模型：很多开源模型（DeepSeek、Qwen、Llama、部分 Meta 模型）有免费档，模型名带 :free 后缀，适合试用和轻量调用。
  • 付费模型：按 token 计费，价格通常跟官方一致或略高——OpenRouter 会收一点服务费作为中转成本。
  • Credit 系统：先充值 credit，调用时按量扣费，比每个厂商分别开账单省心。
  • 速率限制：免费模型有 RPM / TPM 上限；充值后可提高额度。
💡 成本提示

如果你本来就要用 Gemini / Claude 的付费 API，经过 OpenRouter 中转一般只贵一点点（服务费），但换来的是不用每个厂商单独注册 + 单独处理网络问题 + 一套代码切换模型。对国内开发者来说，这笔溢价往往值得。

07其他常见坑

  1. 模型名格式：必须 厂商/模型，不能填官方展示名（比如 Gemini 2.5 Pro 会 404）。
  2. 401 vs 404：401 = key 错；404 = 模型名错（OpenRouter 对未知模型返回 404，不是 400）。
  3. Key 安全：浏览器端直接调 OpenRouter 需要带 key，暴露在前端有泄露风险，正式项目务必走后端代理。
  4. 思考模式：Gemini / Claude 的 thinking 在 OpenRouter 通过 reasoning 参数控制，不是所有客户端都暴露这个开关。
  5. 流式响应：OpenAI SDK 默认 stream=True，客户端要支持 SSE 才能正常显示逐字输出。

08小结

回到开头那个 502：直连 Google API 失败，多半是网络 / 节点问题，不是你配置错了。与其跟 DNS 污染和流媒体节点较劲，不如把请求路径改成「你 → OpenRouter → Google」。一层统一网关，把 300+ 模型收进一个端点、一个 key，换模型只改名字——开发体验提升明显，国内连通性也稳得多。

代价也很清楚：多一层中转、按 token 略贵、免费模型有额度限制。但对于「想稳定用上 Gemini / Claude，又不想在代理配置上耗一下午」的场景，OpenRouter 是目前最省心的方案之一。

09实测补充：地区限制、充值与直连对比

文章发布后，我当场注册了 OpenRouter、拿了 key、从本地客户端实测了一遍连通性。结果比上面写的"国内更稳"要复杂——这里把带时间戳的真实排查过程记下来，给国内读者避坑。

实测时间线（北京时间 2026-07-11）

时间操作结果
21:50用 OpenRouter key 调 google/gemini-2.5-flash（本机直连）403 not available in your region
21:55同一客户端重试，模型仍为 Gemini依旧 403，与沙箱侧测试结果一致
22:00代理节点切到新加坡再测403 → 402：地区限制解除，但提示 Insufficient credits
22:02确认账户从未充值，理清"免费"误区Gemini 在 OpenRouter 并非免费，需先购 credit

坑一：OpenRouter 也救不了 Gemini 的地区限制

很多人（包括我写上文时）以为"走 OpenRouter 中转就能绕过直连的 502"。实测发现：OpenRouter 是合规网关，不是翻墙代理。它到 Google 后端拿模型时，照样执行 Google 的地区规则。只要账号 / 请求 IP 在中国大陆，Gemini 就返回 403。

⚠️ 哪些模型受影响

受地区限制的是美国厂商模型：Gemini（Google）、Claude（Anthropic）、GPT（OpenAI）、Grok（xAI）。不受限制的是 DeepSeek、Qwen、Kimi、GLM、Llama 等——这些从中国大陆就能直接调（非免费模型同样需充值）。

坑二：Gemini 在 OpenRouter 根本不是免费的

这是最大的认知偏差，也是 22:02 那一步才彻底想明白的。Google AI Studio 给的 Gemini 免费额度 ≠ OpenRouter 上的 Gemini 免费。OpenRouter 是 API 网关，每调一次都要付给 Google 上游费用，所以：

  • Gemini / Claude / GPT 在 OpenRouter 全是付费模型，按 token 计费；
  • OpenRouter 要求账户先购买 credit 才能调用任何付费模型（错误码 402 Insufficient credits. This account never purchased credits）；
  • 只有模型名带 :free 后缀的（如部分 Llama / Mistral）才是真免费，但经常排队、限流，质量也不如 Gemini。

一句话：Gemini 在 Google 那能白嫖，在 OpenRouter 这不能白嫖。切新加坡节点只解决了地区限制，接下来要充钱才能用。

关键发现：如果新加坡能用，直连 Google 可能更香

切到新加坡节点后，OpenRouter 放行了 Gemini（403→402）。这证明新加坡 IP 通过了 OpenRouter 的地区判断。同一套逻辑也适用于直连 Google：

路径地区判断方新加坡节点下费用
客户端 → OpenRouter → GoogleOpenRouter✅ 放行（402 仅因未充值）付费（含中转费）
客户端 → Google 直连Google✅ 同样放行（SG 在允许名单）免费（AI Studio 额度）

也就是说，如果你的目标只是用 Gemini，新加坡节点 + 直连 Google（用自己的 key）就够了，还免费用 AI Studio 额度，不用像 OpenRouter 那样充值。原先直连报 502，是因为旧节点到不了 generativelanguage.googleapis.com 这个域名；换成能正常路由 Google API 的新加坡节点后，直连 + 免费，比走 OpenRouter 更省。

OpenRouter 的真正价值：一个 key 调所有模型 + 自动 fallback + 不用每个厂商单独处理网络。如果你只想要 Gemini，配好新加坡节点直连 Google，往往更简单、更便宜。

给国内读者的建议

  • 只要 Gemini：新加坡（或日 / 美）节点 + 直连 Google，走 AI Studio 免费额度，零成本。
  • 想一个 key 调多家 + 自动容灾：用 OpenRouter，但先把模型换成不受地区限制的 DeepSeek / Qwen / Kimi / GLM，且先小额充值再测。
  • 非要 OpenRouter 上的 Gemini / Claude：需同时满足「充钱」+「请求 IP 在允许地区」。新加坡节点可解地区，社区还验证过 Cloudflare AI Gateway 反代、欧洲 VPS 单独给 OpenRouter 走规则两种办法。
📌 本文立场更新

第 03 节说"OpenRouter 帮国内绕开直连"对连通性成立，但对Gemini / Claude 这类美国模型不成立——它们额外受地区限制。本文不删旧内容，保留完整排查轨迹，方便对照自己卡在 403 还是 402。

10 再实测：API 路径返回 JSON，但 Pro 系列不免费

22:09，把浏览器地址从根路径 generativelanguage.googleapis.com 换成完整 API 路径：

https://generativelanguage.googleapis.com/v1beta/models?key=你的GoogleKey

返回了完整的模型列表 JSON——包含 models/gemini-2.5-flash、models/gemini-2.5-pro、models/gemini-3.1-flash-lite 等几十条记录。这说明：

  • 新加坡节点 + 直连 Google API，网络已通；
  • API Key 有效；
  • 地区限制已放行。

之前浏览器访问根路径报 404，是正常行为（API 域名没有网页首页），不是"打不开"。

免费用户该用什么模型

从配额表看，可用且额度充足的模型是 Flash / Flash Lite 系列：

  • gemini-2.5-flash：5 RPM / 250K TPM / 20 RPD，日常开发首选
  • gemini-2.5-flash-lite：10 RPM / 250K TPM / 20 RPD，更便宜
  • gemini-3.1-flash-lite：15 RPM / 250K TPM / 500 RPD，免费层额度最多

注意：OpenAI 兼容端点填写模型名时，要把列表里的 models/gemini-2.5-flash 去掉 models/ 前缀，写成 gemini-2.5-flash。

最终可用配置（直连 Google，免费）

  • Base URL：https://generativelanguage.googleapis.com/v1beta/openai
  • API Key：Google AI Studio Key
  • 模型名：gemini-2.5-flash（或 gemini-3.1-flash-lite）

到此，OpenRouter 充值方案可以暂不执行。对于"只要 Gemini"这个需求，新加坡节点 + 直连 Google + Flash 系列免费模型已经够用。

12 修正：OpenAI 兼容层没有 /models 端点

22:28，按第 11 节的建议访问 https://generativelanguage.googleapis.com/v1beta/openai/models?key=...，Google 返回 404：

"error": { "code": 404, "message": "Requested entity was not found.", "status": "NOT_FOUND" }

这说明 Google 的 OpenAI 兼容层只实现了 /chat/completions，没有实现 /models。所以没法像原生 API 那样列模型清单来查名字，只能直接调用 chat/completions 试错。

修正后的验证方法

用 curl 或客户端直接发一条最小请求，逐个模型名测试：

curl -X POST https://generativelanguage.googleapis.com/v1beta/openai/chat/completions \
  -H "Authorization: Bearer 你的GoogleKey" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "gemini-3.1-flash-lite-preview",
    "messages": [{"role": "user", "content": "ping"}],
    "max_tokens": 10
  }'

如果返回正常回复，说明模型名可用；如果报错，再换下一个。注意：gemini-2.5-flash 返回的 404 不是"路径不存在"，而是"模型对新用户不可用"，这反而证明 /chat/completions 本身是通的。

建议按这个顺序试

  1. gemini-3.1-flash-lite-preview（免费层额度最多：15 RPM / 500 RPD）
  2. gemini-3.5-flash（稳定版，5 RPM / 20 RPD）
  3. gemini-3-flash（备选，5 RPM / 20 RPD）
📌 针对第 11 节的修正

第 11 节建议用 /v1beta/openai/models 查看兼容层支持的模型名，但经实测该端点不存在。正确做法是直接调用 /v1beta/openai/chat/completions 试模型名。本文不删改旧内容，只在此补充。
